## [Unreleased]

### Added
- **Multi-CLI support**: native integration with five coding-agent CLIs — Claude Code (`claude`), Codex (`codex`), Gemini CLI (`gemini`), Antigravity (`agy`), and GitHub Copilot CLI (`copilot`). Each has its own argument builder, output parser, and session model.
- **Cross-platform runners**: PowerShell `limitshift.ps1` (Windows PowerShell 5.1+) and Bash `limitshift.sh` (macOS/Linux, Bash 3.2 compatible). Same flags, same JSON schema, same behavior.
- **Public queue schema**: `limitshift-queue.json` (with `limitshift-queue.schema.json` as the source of truth) defines an ordered task list; each task names a `cli`, `projectPath`, `prompt`, and optional `model`, `effort`, `extraArgs`. A per-queue `settings` block configures retries, limit waits, model rotation, and validation policy.
- **`--validate-only` / `-ValidateOnly`**: parses the queue, validates CLI binaries are on `PATH`, runs model-name discovery, and prints a one-line summary per task — without running anything.
- **`--dry-run` / `-DryRun`**: walks the queue and prints each assembled command at column 0 (greppable) without running any CLI.
- **Multi-queue / state isolation**: each queue file has its own `.limitshift-<name>/` state folder (sessions, outputs, status markers, transcripts, runs.csv). `--queue-path` / `-QueuePath` accepts an absolute path, a relative path with separators (resolved from cwd), or a bare filename (resolved next to the script). Run multiple queues in parallel from one repo.
- **Concurrency lock**: a `limitshift.lock` file in the state folder holds the runner's PID. A second runner pointed at the same queue exits with a clear error naming the queue and live PID. Stale locks (dead PID) are silently cleared.
- **Multi-line prompts via stdin**: prompts go to every stdin-capable CLI through stdin instead of as a process argument, so newlines survive Windows' `Start-Process` and the npm `.cmd` shim. `agy` and `copilot` need `-p`; they receive the prompt via that argument with an empty/EOF stdin so they cannot block on an inherited handle. On Windows, native-exe arguments are delivered as a single canonical command-line string so multi-line prompts and spaces in paths survive `CreateProcess` intact.
- **Structured per-CLI output parsing**: each CLI's JSON / JSONL output is parsed for the final assistant reply, the session id (used for resume), and limit/error signals — no console scraping. `agy`, which has no headless output mode, has its reply recovered from its on-disk conversation transcript (`~/.gemini/antigravity-cli/...`); overridable via `LIMITSHIFT_AGY_DATA_DIR`. `copilot`'s JSONL parser drills into the `data.*` payload so the assistant reply is extracted cleanly.
- **Resume preserves the original task**: on resume the runner repeats the original prompt verbatim alongside the "continue where you stopped" instruction, so a thin first run or a fresh session always has the full task to work from — `/goal …` lines and any prompt-dependent context survive cleanly.
- **Completion-marker checking**: in completion-check mode (default) each prompt has `[[TASK_COMPLETE]]` / `[[TASK_BLOCKED]] <reason>` instructions appended; the runner keeps resuming the task until the marker appears on the last non-empty line. Detection is lenient — the line just needs to contain the marker. `settings.completionCheck: false` (per-task overridable) switches to "simple mode": the prompt is sent verbatim and the first OK run marks the task done.
- **No-progress stall guard**: in completion-check mode, after `settings.maxStalls` (default 2) consecutive identical no-marker responses the task is failed instead of looping all the way to `maxRunsPerTask`.
- **Per-task model rotation**: `tasks[].model` accepts a single name or an ordered list. On a usage limit the runner switches to the next model immediately (same session) and only waits for a reset once every listed model is exhausted, then restarts from index 0. The current index is persisted per task.
- **Per-CLI effort validation**: `effort` is checked against per-CLI rules at queue load — `gemini` requires `null`; `claude` accepts `low|medium|high|xhigh|max` (and rejects effort entirely for haiku models); `codex` accepts `minimal|low|medium|high|xhigh`.
- **Dynamic model validation**: at `--validate-only`, the runner queries each CLI's model-list subcommand and checks each task's configured `model` against the live list. Discoverable today: `agy`. Non-discoverable: `claude`, `codex`, `gemini`, `copilot` — printed as INFO and never blocked. Levenshtein-based typo suggestions when a model isn't found (threshold 4). Policy via `settings.modelValidation`: `"strictWhenDiscoverable"` (default), `"warn"`, or `"off"`.
- **Capability cache**: discovered model lists are persisted under `.limitshift-<name>/capabilities/<cli>.json`. TTL via `settings.capabilityCacheHours` (default 24, set 0 to always refresh). `--refresh-capabilities` / `-RefreshCapabilities` forces a re-query.
- **`--probe-models` / `-ProbeModels`**: opt-in connectivity probe that runs a tiny non-editing prompt per unique CLI during `--validate-only` only. Also via `settings.probeModels: true`.
- **Local models via Ollama**: a task can target a local Ollama model by setting `model` to the model name and adding `["--oss", "--local-provider", "ollama"]` to `extraArgs`. `codex` passes the flags through natively; `claude` (which has no native Ollama flag) is invoked via `ollama launch claude --model <model> --yes -- <claude args>`. A local `claude` task requires a `model` (enforced at validation) and skips the cloud usage check.
- **Fingerprint-based done invalidation**: every `.done` / `.failed` marker stores a SHA-256 fingerprint of the task definition (name, cli, projectPath, model, effort, prompt, extraArgs). Editing any of those auto-invalidates the marker and drops the stale session id, so the task re-runs without manual cleanup.
- **Self-documenting state folder**: each `.limitshift-<name>/` ships with a generated `_README.txt` explaining the layout and how to re-run, a `runs.csv` with one row per CLI run (timestamp, task, run, mode, exit, status), and per-task output files named `task-NN-<slug>-output.txt`. Output files are UTF-8 without BOM so they are grep-able and parseable.
- **Magenta-accent preview UI**: per-task header (`▸ Task N/T · name   cli · model · new`) with a quoted 2-line prompt preview, animated working spinner whose frames go to `/dev/tty` so they stay out of captured logs, past-tense usage-limit beat (`☾ Hit a usage limit on <cli>, rested for <duration>`) with a moon countdown, dim soft separator between tasks, a colored response header, and a >24h guard that refuses to wait through what's likely a weekly reset. Within-run resumes (limit/rest, error retry, no-marker resume) emit a compact one-liner instead of re-printing the full task header.
- **Six-variant "Done" summary**: distinguishes (a) all newly executed and OK, (b) some new + some already-done (skipped from a previous run), (c) some failed under `stopOnError:false`, (d) nothing to do because every task was already done, (e) `--dry-run` complete, and (f) `completionCheck: false` where the runner can't verify what got done — each with the appropriate green/red breakdown, transcript link, and a "delete the state folder to redo skipped tasks" hint where it makes sense.
- **`--demo` / `-Demo`**: scripted, no-network walkthrough that drives the UI through every state using `limitshift-queue.example-workflow.json`. No CLIs run, no quota spent.
- **`-ShowRawOutput` / `--show-raw`**: prints the full raw JSON / JSONL response from each CLI on the console instead of the 10-line trimmed view, plus the assembled command line.
- **`AGENTS.md` (ask your agent to write the queue)**: ships `AGENTS.md` plus `CLAUDE.md` / `GEMINI.md` compatibility pointers that teach an AI coding tool how to build a valid `limitshift-queue.json` from a plain-language draft — fields, sensible model suggestions per CLI, per-CLI effort rules, and the permission flag needed to edit files.
- **Example queues**: `limitshift-queue.example-simple.json` (one task, required fields only, `completionCheck: false`), `limitshift-queue.example-workflow.json` (a review → fix → verify pipeline), and `limitshift-queue.example-advanced.json` (every optional field). All three pass `--validate-only` as part of the regression suite.
- **Beginner-friendly README + QUICKSTART**: top-down rewrite aimed at people who have never used a terminal — what LimitShift is, why in-app queues stall on usage limits, a first-run walkthrough, the "ask your agent" shortcut, and a real review → fix → verify workflow. Reference material under a `## Reference` heading.
- **Regression test suites**: Pester 5 for PowerShell and a pure-bash harness for the Unix runner — covering schema validation, JSON output parsing, CLI argument construction, multi-line stdin delivery, usage-limit handling and model rotation, multi-queue and lock semantics, fingerprint invalidation, the dynamic model-validation modes, and end-to-end runs against stubbed CLI binaries (currently 159 Pester tests and 81 bash tests, all green).
